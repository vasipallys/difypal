## 8. Validation Rules

Implement these validation categories.

### YAML Validation

* Valid YAML syntax.
* No duplicate keys.
* Required top-level fields.
* Correct scalar/list/map types.
* No invalid indentation.
* No unresolved placeholders unless explicitly allowed.

### Dify DSL Schema Validation

Derived from the Dify source code.

Validate:

* DSL version.
* App metadata.
* App mode.
* Workflow graph.
* Nodes.
* Edges.
* Variables.
* Model config.
* Tool config.
* Knowledge config.
* Environment variables.
* Conversation variables if chatflow.
* Start node.
* End/answer/output nodes.

### Graph Validation

Check:

* Start node exists.
* No orphan nodes.
* No broken edges.
* No missing source/target.
* No invalid cycles unless Dify supports them for that node type.
* Branch nodes have proper branches.
* Iteration/loop nodes are structurally valid.
* Every execution path ends correctly.
* Node IDs are unique.
* Edge IDs are unique.
* Node coordinates are valid for canvas.

### Variable Validation

Check:

* Variable references exist.
* Input variables are defined.
* Output variables are consumed correctly.
* System variables are valid.
* Node output paths are valid.
* Type compatibility is reasonable.
* Prompt templates use valid variables.

### Security Validation

Check:

* No hardcoded API keys.
* No secrets in exported docs.
* Secret variables masked.
* Local logs redact secrets.
* External AI calls require user consent.
* Real Dify execution requires explicit approval.
* Uploaded DSL is treated as untrusted input.
* Code nodes are not executed unsafely.
* Electron uses secure defaults.

### Compatibility Validation

Check:

* Dify version compatibility.
* Deprecated fields.
* Missing fields.
* Plugin/model compatibility.
* Knowledge references that cannot be exported.
* Secrets excluded from export.
* Import warnings.
